## 8.1 决策日志（ADR 风格）

### ADR-001：单 crate 作为 Pinocchio 透传入口
- 决策：`geppetto` 保持单 crate，核心依赖 `pinocchio`，CPI helper 按 feature 细分。
- 理由：降低迁移成本，便于 agent 在依赖层使用统一入口。
- 风险：feature 组合复杂度上升；需依赖矩阵稳定管理。

### ADR-002：`AccountSchema` 用 trait+const，而非 derive 宏
- 决策：账户布局通过 trait 常量定义，避免宏生成隐藏逻辑。
- 理由：符合 agent 可读性与可审计性目标，方便静态对齐。
- 风险：调用方需要手工填充布局常量，易出字段偏移错误。
- 缓解措施：通过 `assert_account_size!` 宏和 doctest 示例降低错误概率。

### ADR-003：知识模块采用带时间戳版本头
- 决策：所有知识文档要求 `geppetto`/`pinocchio`/日期三段头。
- 理由：显式告知知识新鲜度，避免过期经验误导生成代码。
- 风险：版本未及时更新会触发 agent 侧验证负担。

## 8.2 复杂度预算（初版）

- 维护复杂度（当前目标）：低到中，集中在三个核心点
  - `guard.rs` 分支覆盖与边界条件
  - `schema.rs` 的布局正确性和验证逻辑
  - `testing.rs`/TS 对齐链路的长期可运行性
- 预算阈值：每次新增功能不得增加 > 3 个公开 API 且必须同步补齐测试规格

## 8.4 演化治理规则

- 任何关键逻辑变更都需先更新 `docs/03-technical-spec.md` 再改代码。
- 任何新增错误码必须同步更新：
  - `error.rs` 常量映射
  - 相关测试
  - 对应知识文档说明
- 代码与文档版本戳必须同一发布节奏更新，不允许只改其中一端。

## 8.5 退化与回滚原则

- 如果出现 `AccountSchema` 或 `assert_pda` 逻辑回归，优先回滚至上一个通过 `cargo test --doc` 的 commit。
- 若版本头/知识头出现不一致，不可进行新发布，必须先同步版本与验证时间戳。
- 遇到高风险安全变更（签名、owner、PDA、rent）优先走离线评审并冻结合并。
